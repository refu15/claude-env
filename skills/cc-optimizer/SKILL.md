# CC-Optimizer: トークンコスト最適化アドバイザー

## When to use

以下の場面で使う：

- Claude Code のトークン消費が多いと感じるとき
- 月のAPI費用を削減したいとき
- CLAUDE.md が肥大化してきたとき
- プロンプトの書き方を改善したいとき
- モデルの使い分けを見直したいとき

## コスト削減の5戦略

### 戦略①: プロンプトを具体的に書く（最大10倍の差）

**悪い例（曖昧）:**
```
このコードベースのバグを直して
```
→ Claude が全ファイルを読み漁る。トークン爆発。

**良い例（具体的）:**
```
`auth.ts` の `validateToken` 関数で、期限切れトークンが正しく弾かれないバグがある。
72行目あたりの条件分岐を修正して
```
→ ピンポイントで読む。消費最小化。

**プロンプト作成チェックリスト:**
```
[ ] どのファイルか？ → ファイル名を明示
[ ] どの関数/クラスか？ → 関数名・クラス名を明示
[ ] 何行目か？ → 行番号を明示（わかる場合）
[ ] 期待する動作は？ → 出力形式・条件を明示
[ ] スコープは？ → 変更範囲を限定
```

### 戦略②: コンテキスト管理（/compact と /clear）

**タスクが切り替わったら → `/clear`**
```
# バグ修正が終わったら
/clear
# 新機能の設計へ（クリーンな状態で開始）
```

セッション保存してから切る：
```
/rename bug-fix-auth-2026-03
/clear
```
後で戻す場合は `/resume`

**会話が長くなってきたら → `/compact`**
```
/compact コードの変更点とAPIの使い方だけ残して
```

**コンテキスト管理の目安:**
| 状況 | 推奨アクション |
|------|--------------|
| タスクが切り替わった | `/clear` |
| 会話が50往復を超えた | `/compact` |
| 無関係な情報が溜まった | `/compact [残す内容を指定]` |
| セッションを完全にリセット | `/clear` |

### 戦略③: CLAUDE.md をスリムに保つ（500行以下）

CLAUDE.md はセッション開始時に **毎回** 読み込まれる = 全メッセージのトークンに課金される。

**理想的な構造:**
```
CLAUDE.md（常時読み込み・500行以下）
├── プロジェクト概要（5行）
├── ディレクトリ構成（10行）
├── 基本コーディング規約（20行）
└── よく使うコマンド（10行）

skills/（オンデマンド読み込み）
├── デプロイ手順 → deploy.md
├── レビューガイドライン → review.md
├── テスト戦略 → testing.md
├── API仕様 → api-spec.md
└── ...
```

**肥大化チェック:**
```bash
wc -l CLAUDE.md
# 500行超えたらスキルへの分離を検討
```

### 戦略④: モデルを使い分ける

| モデル | 使いどころ | コスト感 |
|--------|-----------|---------|
| **Haiku** | サブエージェント、単純処理、フォーマット変換 | 最安 |
| **Sonnet** | 日常コーディング、修正、調査（デフォルト） | 安い |
| **Opus** | 複雑設計、アーキテクチャ判断、難解なバグ | 高い |

**切り替え方:**
```
/model claude-haiku-4-5    # 単純タスクへ
/model claude-sonnet-4-6   # 通常作業へ戻す
/model claude-opus-4-6     # 重要な判断のみ
```

**推奨フロー:**
```
情報収集・調査    → Sonnet
設計判断・アーキ  → Opus（ここだけ）
実装・コーディング → Sonnet
テスト生成        → Haiku または Sonnet
```

**Opus の拡張思考コスト注意:**
- デフォルト思考バジェット: 約32,000トークン
- 出力トークンは入力の5倍のコスト
- 単純タスクには思考をオフにする

### 戦略⑤: 重い処理をサブエージェントに委譲

大量出力を本体セッションでやると、その後の全メッセージに課金され続ける。

**やってはいけないパターン:**
```
# テスト結果10,000行がコンテキストに入り込む
run all tests and show me every output line
```

**正しいパターン（サブエージェントに委譲）:**
```
テストをサブエージェントで実行し、失敗したテストのファイル名と行番号だけを報告して
```
→ サブエージェントの独立コンテキストに大量ログが閉じ込められ、本体には要約だけが返る。

**フックでフィルタリングする方法:**
```bash
# テスト実行時に失敗のみ抽出するフック
# PostToolUse に設定して成功ログをカット
```

**委譲すべき処理の例:**
```
- テスト実行（大量ログ）
- ログファイル解析
- 大規模ドキュメント取得
- ビルド出力の確認
- 依存関係ツリーの展開
```

## コスト見積もりとモニタリング

Anthropic の公式データ:
- 平均コスト: 約$6/日（約900円）
- 90%のユーザー: $12/日以下
- 月換算: $100〜200

ガチ使いで超えてくる場合、上記5戦略で30〜70%削減可能。

## クイック診断

現在のセッションで改善できることを確認：

```
1. 今のプロンプトに「このファイルの」「全体的に」などの曖昧表現が含まれていないか？
2. タスクが切り替わってから /clear を打ったか？
3. CLAUDE.md は500行以下か？
4. 今やっているタスクに Opus が本当に必要か？
5. テスト実行やログ解析をメインセッションでやっていないか？
```
