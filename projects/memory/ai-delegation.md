# AI 委譲ルール（Codex / Gemini / Claude）

## 概要
Claude はオーケストレーターとして、タスクの性質に応じて Codex・Gemini を呼び出す。

## 呼び出しスクリプト

```bash
# Codex（コード特化）
/home/nnkre/.claude/scripts/ask-codex.sh "プロンプト"

# Gemini（長文・調査・マルチモーダル）
/home/nnkre/.claude/scripts/ask-gemini.sh "プロンプト"
```

---

## ルーティングテーブル

### → Codex に委譲する

| トリガー | 例 |
|--------|-----|
| シンプルなコード生成 | 「〜する関数を書いて」「〜コンポーネントを作って」 |
| リファクタリング | 「このコードを整理して」「DRYにして」 |
| バグ修正（単発） | 「このエラーを直して」 |
| テストコード生成 | 「このAPIのテストを書いて」 |
| コードの説明 | 「このコードが何をしているか説明して」 |

**条件**: 対象ファイルが明確 かつ 推論・設計が不要なとき

### → Gemini に委譲する

| トリガー | 例 |
|--------|-----|
| 長文の要約 | 文書・仕様書が1000字超 |
| リサーチ・調査 | 「〜について調べて」 |
| ドキュメント生成 | README・仕様書・説明文の作成 |
| 比較・まとめ | 複数の選択肢を整理して説明 |
| 画像・ファイル分析 | マルチモーダル系タスク |

**条件**: コードより文章・情報処理がメインのとき

### → Claude 自身が担当する

| タスク | 理由 |
|--------|------|
| アーキテクチャ設計 | 複合的な判断が必要 |
| エージェント間の調整・統合 | オーケストレーション役 |
| セキュリティレビュー | 慎重な推論が必要 |
| 複数ステップの複雑なタスク | 文脈保持が必要 |
| Codex/Gemini の出力統合 | 最終まとめ役 |

---

## 実行パターン

### パターン1: 単純委譲
```
ユーザー要求 → Codex/Gemini → 結果をそのまま返す
```

### パターン2: 並列委譲
```
ユーザー要求 → Codex（コード生成）
             → Gemini（仕様文書生成）
             → Claude が統合して返す
```

### パターン3: 直列パイプライン
```
ユーザー要求 → Gemini（調査・設計）→ Codex（実装）→ Claude（レビュー）
```

---

## 注意事項

- Codex の出力には token 数等の余分な行が含まれることがある → grep でフィルタ
- Gemini の出力には認証ログ行（`Loaded cached credentials.`）が含まれる
- 両CLIともタイムアウトが長くなる場合がある（最大60秒想定）
- 作業ディレクトリを指定する場合は `WORK_DIR=/path` 環境変数を使う
